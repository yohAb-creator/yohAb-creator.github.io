    Sparse vs Dense Retrieval (and why you want both)

    Sparse retrieval (BM25, etc.) matches on actual words/tokens. Great when the query contains the right terms (units, alloys, “800°C”, specific standards). Bad when the wording drifts.

    Dense retrieval (embeddings) matches on meaning. Great when the user says it differently than the doc. Bad when it pulls things that are “vaguely about the topic” but not the exact thing.

    So:

    • Sparse = precise but brittle
    • Dense = flexible but fuzzy

    A hybrid RAG setup runs both, then fuses the results (RRF / weighted merge). You get:

    • docs that actually mention the thing, and
    • docs that mean the thing.

    That’s how you stop the model from saying, “I couldn’t find 800°C, so here’s a random metallurgy paragraph.”

    I’m testing this pattern in a side project; if you handle technical docs, this is relevant.
